Ниже — идеи в духе того, что у вас уже есть (память, типы, вики-линки), без обязательной реализации.

Документация и связность
see / also — явные перекрёстные ссылки между типами (see CPhysical), чтобы строить граф и проверять «битые» имена.
Версия/ревизия образа — image 0x12345678 или build "steam" рядом с module, чтобы не смешивать адреса разных билдов.
Условные блоки — #if NTSC / #region только для генерации доков, не для семантики.
Синтаксис типов и полей
Именованные «слоты» битов уже частично через bitfield; можно inline-битовые поля в одной строке для узких масок без отдельного top-level.
alignas / паддинг — явный padding 0x2 или align 4, чтобы ловить дыры в раскладке.
Константы — const MAX_PEDS 140 в .re и подстановка в комментариях/HTML.
Нативы и сигнатуры
stdcall / thiscall или calling convention как метаданные (даже если только для экспорта в IDA/Ghidra скрипты).
Группировка по VMT — секция vtable CEntity { 0x0 Foo; ... } как «документный» слой поверх сырых оффсетов.
Валидация и качество
Правила линтера в project.toml: «все static должны иметь source», «имя типа PascalCase», дубликаты адресов между файлами.
Дифф адресов между двумя снимками проекта (для обновления после патча).
Инструменты
LSP (hover на тип, go-to-definition по имени типа) — раз у вас уже есть IR и HTML.
Экспорт в JSON/YAML одной командой — для скриптов на Python без парсинга .re руками.
Что не тащить без нужды
Полноценные выражения и операторы — вы сами правильно отсекли; это быстро превращает DSL в мини-C с багами.
Сложный препроцессор — дорого поддерживать; лучше генерация из внешнего конфига.
Если скажешь приоритет (доки vs валидация vs тулинг), можно сузить до 2–3 пунктов с максимальной отдачей. В Agent mode могу помочь внедрить что-то одно по шагам.
